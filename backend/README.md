# Подсистема интеграции со СВАП (SVAP Query Service)

Данная подсистема предназначена для получения данных из внешней системы СВАП и интеграции со справочниками функциональных подсистем (ФП).

## Архитектура

Подсистема реализована на языке Go и состоит из следующих модулей:

- `internal/model`: Общие структуры данных (Аналитические признаки, Заголовки СВАП).
- `internal/config`: Динамическая конфигурация с поддержкой групп и синхронизации между узлами.
- `internal/svap`: Клиент для взаимодействия со СВАП (протоколы HTTP).
- `internal/repository`: Интерфейсы для хранения АП, кэширования справочников и конфигурации.
- `internal/service`: Бизнес-логика, включая периодическую синхронизацию АП (раз в сутки).
- `internal/api`: REST API хендлеры для управления запросами и конфигурацией.

## Структура базы данных (PostgreSQL)

Сервис использует PostgreSQL для хранения конфигурации, кэша и результатов. Основные таблицы:

- `app_config`: Динамические настройки, разделенные по группам (`server`, `svap`).
- `analytical_attributes`: Справочник всех АП, доступных в СВАП, с правилами их валидации (регэкспы, сервисы).
- `dictionaries_cache`: Локальный кэш справочников функциональных подсистем (ФП) для ускорения работы UI.
- `saved_queries`: Реестр поисковых запросов пользователей (параметры фильтрации, тип сервиса).
- `query_results`: Хранилище результатов выполнения запросов (витрины данных), которые могут быть переданы в подсистему АИ.

SQL-скрипты инициализации расположены в `db/migrations/`. 

При старте сервис автоматически выполняет все новые миграции, отслеживая их состояние в таблице `schema_migrations`. Для обеспечения безопасности в многоузловой среде используется **Postgres Advisory Lock**, который гарантирует, что только один экземпляр будет выполнять миграции в конкретный момент времени, предотвращая конфликты. Это гарантирует, что база данных всегда находится в актуальном состоянии на всех узлах.

## Динамическая конфигурация

Система поддерживает изменение настроек "на лету" без перезапуска сервиса.

### Группы настроек
1. **server**: Настройки HTTP-сервера (порт).
2. **svap**: Адреса хостов систем-источников (GK, LS, JO).
3. **retention**: Настройки времени жизни (TTL) витрин данных.
   - Поддерживает иерархию: `user_ttls` -> `role_ttls` -> `org_ttls` -> `default_ttl`.
   - Если задан TTL для конкретного пользователя (`user_ttls`), используется он.
   - Иначе, если у пользователя есть роли с переопределенным TTL (`role_ttls`), выбирается максимальное значение.
   - Иначе, если задан TTL для организации пользователя (`org_ttls`), используется он.
   - В противном случае используется `default_ttl`.

### Синхронизация между узлами
При работе в многоузловом режиме (кластере) используется механизм **Postgres LISTEN/NOTIFY**:
- Изменение конфигурации через любой узел сохраняется в БД.
- БД отправляет уведомление всем остальным узлам.
- Узлы автоматически обновляют локальную память.
- Гарантированность доставки обеспечивается принудительной синхронизацией при старте сервиса и периодической сверкой (polling).

### REST API Управления
- `GET /api/v1/config`: Просмотр всей подготовленной конфигурации.
- `GET /api/v1/config/groups`: Список доступных групп настроек.
- `GET /api/v1/config/groups/{name}`: Просмотр конкретной группы (напр. `server`, `svap`, `retention`).
- `PUT /api/v1/config`: Обновление черновика конфигурации.
- `POST /api/v1/config/apply`: Применение изменений (сохранение в БД и уведомление всех узлов).

## Основные функции

1. **Синхронизация АП**: Сервис запрашивает перечень аналитических признаков из СВАП и сохраняет их в локальном репозитории.
2. **Кэширование справочников**: Для ускорения работы UI и валидации признаков, данные из ФП кэшируются локально. Поддерживается поиск по части кода/наименования.
3. **Выполнение и сохранение запросов**: Подсистема формирует запросы к СВАП (OPLIST, FSG, TURN, ReportGK, COR, PA, CONS) на основе сохраненных параметров, получает данные и сохраняет их в `query_results` в формате JSONB для дальнейшего анализа.
4. **Интеграция с АИ**: Сохраненные витрины данных содержат метаинформацию, необходимую для работы алгоритмов аналитического инструмента.

## Модели данных

### Аналитический признак (AnalyticalAttribute)
- `Code`: Код признака (напр. `BUD_CODE`).
- `Businesses`: Список систем, использующих признак (`ФБ`, `ПУД`).
- `ValidationType`: Способ валидации (Регулярное выражение, Сервис или Список).

## Спецификация запросов СВАП

Запросы к СВАП состоят из заголовка (`header`) и тела (`document`). 
В заголовке передаются технические параметры:
- `Subject`: Инициатор события.
- `Offset/Limit`: Параметры пейджинга.
- `SettlementDate`: Дата отражения.

## Запуск

Для сборки проекта с учетом версии Git:
```bash
cd backend
go build -ldflags "-X 'svap-query-service/backend/internal/version.GitCommit=$(git rev-parse --short HEAD)' -X 'svap-query-service/backend/internal/version.GitBranch=$(git rev-parse --abbrev-ref HEAD)'" -o main ./cmd/server
```

Или через Docker:
```bash
docker build --build-arg GIT_COMMIT=$(git rev-parse --short HEAD) --build-arg GIT_BRANCH=$(git rev-parse --abbrev-ref HEAD) -t svap-query-service .
```
